## Decisions & deviations

- `Textarea` shadcn component was not pre-installed; added via `npx shadcn@latest add textarea` for the registration form description field.
- Registration sends a `SEND_GYM_PENDING` notification email (no magic link). The magic link is only created and sent when a SUPER_ADMIN calls `PATCH /gyms/:id/approve`. This enforces the approval gate end-to-end.
- `requestMagicLink` silently discards requests from ORG_ADMIN users whose gym is not `ACTIVE` (consistent with the non-existent-email pattern — no info leakage).
- `verifyMagicLink` throws `403 Forbidden` if the user is an ORG_ADMIN with a `PENDING`, `REJECTED`, or `SUSPENDED` gym — safety net for stale tokens.
- `PATCH /gyms/:id/reject` and `PATCH /gyms/:id/suspend` require a mandatory `reason` body field (Zod-validated); the reason is stored in `Gym.statusReason` and returned on `GET /gyms/:id`. Schema extended via migration `20260622133920_add_gym_status_reason`.
- `PATCH /gyms/:id/suspend` also immediately terminates all active sessions for the gym owner via `SessionService.deleteAllUserSessions`.
- `PATCH /gyms/:id/reactivate` sets status back to `ACTIVE`, clears `statusReason`, and sends the owner a fresh magic link.
- `description` on gym registration is limited to 500 characters; `reason` on reject/suspend actions is limited to 500 characters. Both limits are enforced via Zod `.max()` in the shared contracts package (hits NestJS `ZodValidationPipe` server-side and `zodResolver` client-side). Frontend shows a live `X / 500 characters` counter that turns red when the limit is exceeded.
- `phone` (min 7, max 20 chars) and `address` are mandatory on gym registration — added to `Gym` schema, contracts, `GYM_DETAIL_SELECT`, and the registration form.
- Gym `name` is `@unique` in the DB. `register()` catches Prisma `P2002` and throws `ConflictException` (409) so the frontend `ApiError` surfaces a readable message.
- **A3 subscriptions panel extracted to co-located component:** `subscriptions-panel.tsx` is a separate file next to `members/[id]/page.tsx` (not inlined) to keep the page file manageable. `SubscriptionsPanel`, `AddSubscriptionDialog`, and `CancelSubscriptionDialog` all live in `subscriptions-panel.tsx`; `page.tsx` just imports and renders `<SubscriptionsPanel memberId={memberId} />`.
- **A3 `SubscriptionsController` uses `@Controller()` (empty prefix):** routes are `GET /members/:memberId/subscriptions`, `POST /subscriptions`, `PATCH /subscriptions/:id/cancel` — the member-scoped GET lives in the subscriptions module (not the members module) to keep domain logic together. NestJS resolves the `/members/:memberId/subscriptions` path unambiguously over `/members/:id` because the extra path segment differentiates them.
- **A4 role cookie for middleware redirect:** `auth.controller.ts` sets a non-HttpOnly `bootcamp_starter_role` cookie alongside the session cookie on login; cleared on logout. `proxy.ts` reads it to redirect MEMBER → `/portal` and block cross-role access without an extra API call in the middleware.
- **A4 portal route group is `app/(member)/`:** portal pages live at `/portal`, `/portal/subscriptions`, `/portal/plans`, `/portal/profile`. The `(member)` segment is invisible in the URL. `MemberSidebar` is a separate component (not the admin `AppSidebar`).
- **A4 invite edge cases:** `POST /members/:id/invite` rejects if (1) member not found/not in gym → 404, (2) `member.userId` already set → 409 "already invited", (3) a User with the member's email already exists → 409 "email conflict". User provisioning and `Member.userId` link happen inside a `$transaction`.
- **A4 member deactivation — login block + deactivated screen:** `requestMagicLink` silently returns `{ success: true }` for MEMBER users with INACTIVE member status (no email sent); `verifyMagicLink` throws `403 ForbiddenException('Your account has been deactivated')` for INACTIVE members — blocks new logins. For already-logged-in members, the session is kept alive (mirrors gym-suspension pattern — no session kill); `auth.service.ts` `getMe()` now includes `member { status }` in the Prisma query and exposes `memberStatus` in the `/auth/me` response; `UserResponse` contract gains an optional `memberStatus` field. `useUser()` detects `memberStatus === 'INACTIVE'` and redirects to `/portal/deactivated` (same pattern as `gymStatus === 'SUSPENDED'` → `/suspended`); `/portal/deactivated` added to `AUTH_ROUTES` to prevent redirect loops. New page `app/(member)/portal/deactivated/page.tsx` shows "Account Deactivated" message with logout button, styled identically to the suspended page.
- **A4 global 401 redirect in SWRProvider:** Added `onError` to the root `SWRConfig` in `lib/swr-provider.tsx` — any SWR hook that receives a 401 immediately calls `router.replace('/login?redirect=...')`, skipping the auth-route guard to prevent loops. Covers natural session expiry across the whole app.
- **A4 `MePortalModule` portal resolution:** all `/me/*` endpoints resolve the caller's `Member` via `prisma.member.findFirst({ where: { userId: user.id, gymId } })` — never trusts a client-supplied memberId.
- **A2 post-merge: plan name unique per gym** — added `@@unique([gymId, name])` to `MembershipPlan` (migration `20260624000000_add_membership_plan_name_unique`); service catches Prisma `P2002` → `ConflictException` 409; both `AddPlanDialog` and `EditPlanDialog` catch 409 and call `form.setError('name', ...)` for inline field error. DB was reset during this work to resolve a pre-existing migration drift (`20260622211213` had been edited after being applied).
- **Pagination pattern (established in A1/A2, must be followed in all future list pages):** export a named `*_PAGE_SIZE = 25` constant from the hook file; pass it explicitly as `limit` in the URL params; use the same constant for `totalPages` math and the "Showing X–Y of Z" range text. Never rely on the backend default. Applies to: A1 Members (`MEMBERS_PAGE_SIZE`), A2 Plans (`PLANS_PAGE_SIZE`). Future list pages (B0 Instructors, B3 My bookings) must follow the same pattern.
- **Inline form validation pattern (established in A2):** all forms use `mode: 'onTouched'` (errors appear after first blur, then on every change). Always add `noValidate` to the `<form>` element — without it, the browser fires its own native tooltip ("Please enter a valid value") on submit before Zod runs, bypassing RHF error display. For `type="number"` inputs, always pass `{ valueAsNumber: true }` to `form.register(...)` — without it, RHF passes a string to Zod and inline validation misfires. Pair with `z.coerce.number()` in the Zod schema so the resolver handles both string and number inputs safely. Applies to every form with number or email inputs in future features (B0 instructor forms, B1 session capacity/times, etc.).
